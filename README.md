
# リポジトリのフォークから日々の記録までの操作手順

このリポジトリを使った演習の進め方と、GitHub への学習記録の残し方をまとめます。

> **はじめての用語が出てきたら**
> この手順書では「フォーク」「クローン」「ステージング」「コミット」「プッシュ」といった Git / GitHub の用語が登場します。意味がわからなくなったら、文末の **[用語集](#用語集)** をいつでも参照してください。

---

## 1. リポジトリをフォークする

まず、このリポジトリを自分の GitHub アカウントに **フォーク（fork）** します。

> **フォークとは？**
> 他の人の GitHub リポジトリを、**自分のアカウント配下にまるごとコピーする**操作です。コピー後はあなた専用のリポジトリになるので、自由に変更してもオリジナル（元のリポジトリ）には影響しません。教材リポジトリを自分の作業場として使うために、まず最初に行います。

## 3. リポジトリをクローンする

ログイン後、フォークしたリポジトリを **クローン（clone）** します。

> **クローンとは？**
> GitHub 上にあるリポジトリを、**自分の PC（ローカル環境）にダウンロードしてくる**操作です。フォークが「GitHub 上での自分用コピー作成」だとすれば、クローンは「そのコピーを自分の PC に持ってくる」イメージです。クローンしてはじめて、VSCode で実際にコードを編集できるようになります。

<img width="903" height="620" alt="リポジトリのクローン" src="https://github.com/user-attachments/assets/8322a42d-aff4-449e-8fe8-864372c3f9b5" />

クローンしたいリポジトリを選びます。次に、PC 上のどこにリポジトリ（フォルダ）を保存するかを指定してください。

> **保存場所の推奨**
> どこでも構いませんが、一般的には **C ドライブ直下** の忘れない場所に `git` などのフォルダを作成し、そこに保存するようにすると覚えやすいです。

<img width="897" height="365" alt="保存場所の指定" src="https://github.com/user-attachments/assets/f5fa3745-5087-4685-8595-48434cdf9ee0" />

> **「保存先フォルダ」と「リポジトリのフォルダ」を混同しないこと**
>
> ここで指定するのは **「リポジトリを置く親フォルダ」** です。**リポジトリ自体のフォルダではありません。**
>
> たとえば `C:\git` を作成して、そこを保存先に指定した場合、VSCode は **`C:\git` の中に、リポジトリ名と同じフォルダを自動で作成**してクローンします。
>
> **正しい指定**
>
> ```
> C:\git  ← ここを保存先に指定する（親フォルダ）
>   └── リポジトリ名\   ← VSCode がここを自動で作る（実際のリポジトリ）
>         ├── .git\
>         ├── README.md
>         └── ...
> ```
>
> **よくある間違い**
> 先に `C:\git\リポジトリ名` というフォルダを自分で作って、そのフォルダを保存先に指定してしまうと、`C:\git\リポジトリ名\リポジトリ名\...` のように **二重のフォルダ** ができてしまいます。
>
> ```
> C:\git\
>   └── リポジトリ名\        ← 自分で先に作ってしまったフォルダ
>         └── リポジトリ名\  ← VSCode がさらにこの中に作ってしまう
>               ├── .git\
>               └── ...
> ```
>
> **保存先には、リポジトリ名のフォルダを含まない `C:\git` を選んでください。** リポジトリ名のフォルダは VSCode が自動で作ってくれます。

## 5. 変更を GitHub に保存する（ステージング → コミット → プッシュ）

演習内容を GitHub に保存して日々の記録をつけていく場合、変更したファイルが「ソース管理」に表示されます。

> **GitHub に保存するまでの 3 ステップ**
> 編集した内容は、**ステージング → コミット → プッシュ** という 3 段階を経て GitHub に反映されます。
> それぞれの意味は次のとおりです。
>
> 1. **ステージング（staging）**:「次のコミットに含める変更」を選んでマークしておく作業。すべての変更を一気に保存するのではなく、関連する変更だけをまとめて記録するために、いったん「準備テーブル」に並べるイメージです。
> 2. **コミット（commit）**: ステージングした変更に **メッセージをつけて、ローカル（自分の PC）の Git に記録する**操作。これは「セーブポイント」のようなもので、後から「いつ・何を・なぜ変更したか」を辿れるようになります。**この時点ではまだ GitHub には反映されていません。**
> 3. **プッシュ（push）**: ローカルに溜まったコミットを **GitHub（リモート）に送信して反映させる**操作。これでようやく GitHub 上のリポジトリに変更が現れ、学習記録として残ります。

<img width="1078" height="620" alt="ソース管理の表示" src="https://github.com/user-attachments/assets/682b636a-563f-4671-9a29-bb7beffc22c4" />

### 5-1. 変更をステージングする

変更したファイルの「+」ボタンを押して、コミットに含める変更をステージングします。

<img width="405" height="330" alt="ステージング" src="https://github.com/user-attachments/assets/4f7d2abb-b928-4744-9f9f-c80db2f3564c" />

### 5-2. コミットメッセージを書いてコミットする

コミットメッセージを書いて、コミットボタンを押下します。コミットメッセージは「何を・なぜ変更したか」が後から自分で見てわかるように書くのがポイントです（例：`第3章 演習1 完了`、`ボタンの色を変更`）。

<img width="407" height="324" alt="コミットの実行" src="https://github.com/user-attachments/assets/c0b6b245-ee80-4d7e-bda4-9c770eeabf12" />

> **`user.name` と `user.email` を聞かれた場合**
> 1 年生の時の設定ができていない可能性があります。表示されたダイアログは **キャンセル** してください。
>
> <img width="454" height="147" alt="user.name / user.email のダイアログ" src="https://github.com/user-attachments/assets/a96a5e6b-6b48-444f-8788-a7416e82deb5" />
>
> コマンドプロンプトを開いて、以下のコマンドで設定してから、もう一度コミットしてください。
>
> ```bash
> git config --global user.name "あなたの名前"
> git config --global user.email "あなたのメールアドレス"
> ```
>
> ここで登録する名前とメールアドレスは GitHub と同じものである必要はありませんが、**なるべく合わせるとわかりやすい**です。

### 5-3. 変更を同期（プッシュ）する

「変更の同期」を押すと、あなたの GitHub のリポジトリへ push されて完了です。

## 日々の運用について

日々、少しでも作業を行ったら **コミットして同期（プッシュ）** することで、GitHub に学習の記録を残すことができます。こまめに管理してください。

## 用語集

この手順書に登場する Git / GitHub の用語をまとめます。

| 用語 | 英語 | 意味 |
| --- | --- | --- |
| **フォーク** | fork | 他人のリポジトリを **自分の GitHub アカウント配下にコピー**すること。元のリポジトリには影響を与えずに自由に編集できる。 |
| **クローン** | clone | GitHub 上のリポジトリを **自分の PC にダウンロード**してくること。これにより VSCode などで編集できるようになる。 |
| **ステージング** | staging | 編集した変更のうち、**次のコミットに含めるものを選んでマーク**しておく作業。`git add` に相当。 |
| **コミット** | commit | ステージングした変更に **メッセージをつけてローカルの Git に記録**する操作。学習の「セーブポイント」。この時点ではまだ GitHub には送られていない。 |
| **プッシュ** | push | ローカルのコミットを **GitHub（リモート）に送信して反映**させる操作。これで初めて GitHub 上に変更が現れる。 |
| **プル** | pull | （参考）GitHub 側の変更を **自分の PC に取り込む**操作。複数の場所で作業するときに使う。 |
| **リポジトリ** | repository | コードや変更履歴を保管する **「箱」**。GitHub 上のものを「リモートリポジトリ」、PC 上のものを「ローカルリポジトリ」と呼ぶ。 |
